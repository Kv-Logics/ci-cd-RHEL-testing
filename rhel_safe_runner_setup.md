# RHEL Multi-Hosting Runner Deployment Guide

This guide outlines the complete workflow, division of roles, and folder structures for deploying full-stack Docker projects on a RHEL server.

---

## 📁 1. Project Directory Structure (On the Server)

The shared runner package is installed in `/opt/`. The developer's configuration files, environments, and Git code are isolated inside the domain folder:

```text
/var/www/domain-x/                    # Domain Root Space (Access via FileZilla)
├── .env                              # Uploaded via FileZilla (Stores database & API secrets)
│
├── github-runner/                    # Created via FileZilla (Stores connection keys)
│   ├── .runner                       # Connection keys (Generated by config.sh)
│   └── .credentials                  # Credentials (Generated by config.sh)
│
└── project-a/                        # Target Workspace (Git clones code here automatically)
    ├── docker-compose.yml            # Loads environment variables from "../.env"
    ├── frontend/                     # React Frontend folder (contains Dockerfile)
    ├── backend/                      # Node Backend folder (contains Dockerfile)
    └── database/                     # PostgreSQL database folder (contains Dockerfile)
```

---

## 🛠️ 2. Role Division

### 💻 Developer Workflow (via FileZilla/GitHub)
The developer does not need SSH access. They set up the environment and commit the code:
1. **GitHub Repository**: Pushes code containing `docker-compose.yml` and the source files.
2. **FileZilla**: Creates a folder named `github-runner` inside `/var/www/domain-x/`.
3. **FileZilla**: Uploads a single `.env` file to the root of the domain space (`/var/www/domain-x/.env`).
   * *In the Git code, the `docker-compose.yml` is configured to read environment secrets from `../.env` to ensure they are never deleted by Git updates.*

---

### ⚙️ Admin/DevOps Workflow (via SSH Terminal)
The server administrator installs the runner globally once, and runs simple control commands for each domain when requested.

#### Shared Initial Installation (Done ONCE on RHEL):
```bash
sudo mkdir -p /opt/github-runner-base/bin
cd /opt/github-runner-base/bin
sudo curl -o actions-runner.tar.gz -L https://github.com/actions/runner/releases/download/v2.316.1/actions-runner-linux-x64-2.316.1.tar.gz
sudo tar xzf ./actions-runner.tar.gz
sudo chown -R $USER:$USER /opt/github-runner-base
```

#### The 3 Commands for Domain Management:
When a developer requests setup or updates, the admin runs only these three commands:

##### 1. Initialize Keys (Run once per project setup):
```bash
cd /var/www/domain-x/github-runner
/opt/github-runner-base/bin/config.sh --url https://github.com/Kv-Logics/geofence-engine --token GITHUB_RUNNER_TOKEN --work ../project-a
```

##### 2. Start the Runner:
```bash
cd /var/www/domain-x/github-runner
nohup /opt/github-runner-base/bin/run.sh &
```

##### 3. Stop the Runner:
```bash
pkill -f "domain-x/github-runner"
```
